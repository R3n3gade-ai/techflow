1.  Purpose & Scope


The Portfolio Architecture Integrity Engine (PAIE) is an autonomous ARMS module responsible for maintaining portfolio architecture integrity at the moment of deployment queue execution. It operates as a silent pre-execution layer — running every required check, resolving every resolvable condition per the governing rulebook, and executing or adjusting the queue without PM input.

PAIE is the primary operational expression of the Sovereign Execution Mandate (Addendum 5). Its design is governed by one principle:

Core Principle PAIE never asks the PM a question. PAIE reports what it did. It does not request permission to do it. Every output directed at the PM is informational, not decisional.


PAIE does not replace or override CAM, CDF, PDS, or any existing ARMS module. It operates as a pre-execution integrity layer that ensures the aggregate portfolio architecture remains coherent at the moment of deployment — catching conditions that individual modules do not see because they operate on single positions, not on the book as a whole.

2.  SEM Compliance Certification


SEM Compliant: YES PAIE contains no PM interface. PAIE contains no confirmation mechanism directed at the PM. PAIE produces no decision prompts, approval requests, or questions directed at the PM. PAIE output to the PM is read-only: EOD snapshot inclusion only. All PM regulatory touchpoints generated by PAIE are administrative acknowledgments pre-populated with the action taken and the governing rule. They require no analysis.


3.  Module Architecture


PAIE is implemented as a single Python module:

src/engine/paie.py


The module comprises four functional components:

3.1  Thesis Tag Registry
A structured data store mapping every position in the book to one or more thesis anchor tags. Tags are defined by the PM at position intake as a classification statement — not a mathematical decision — and implemented by engineering in rulebook.json.

thesis_tag_registry = {     'GEV':   ['AI_POWER_INFRA', 'GRID_MODERNIZATION'],     'ETN':   ['AI_POWER_INFRA', 'DISTRIBUTION_BOTTLENECK'],     'CEG':   ['AI_POWER_INFRA', 'NUCLEAR_BASELOAD'],     'VST':   ['AI_POWER_INFRA', 'POWER_GENERATION'],     'VRT':   ['AI_POWER_INFRA', 'DATA_CENTER_COOLING'],     'GOOGL': ['AI_PLATFORM',    'HYPERSCALER'],     'NVDA':  ['AI_SILICON',     'HYPERSCALER'],     'PLTR':  ['AI_SOFTWARE',    'DEFENSE_TECH'],     # ... all positions tagged at intake }


3.2  Thesis Concentration Calculator
Calculates real-time thesis exposure weight as a percentage of total equity book NAV. Runs at pre-flight and compares calculated weights against single-thesis concentration caps defined in rulebook.json.

def calculate_thesis_concentration(book, registry, nav):     concentration = {}     for ticker, tags in registry.items():         weight = book.get(ticker, {}).get('nav_weight', 0)         for tag in tags:             concentration[tag] = concentration.get(tag, 0) + weight     return concentration   # { 'AI_POWER_INFRA': 0.312, ... }  # Example rulebook cap: SINGLE_THESIS_CAP = 0.30   # 30% of equity book per thesis tag


3.3  Sizing Coherence Engine
When two or more positions share a thesis tag and are subject to simultaneous deployment, validates that their relative sizing is consistent with hierarchy rules in rulebook.json. Inconsistencies are resolved automatically. The resolution is logged. The queue is adjusted. Execution proceeds.

def check_sizing_coherence(queue, registry, rulebook):     adjustments = []     pairs = find_shared_thesis_pairs(queue, registry)     for pair in pairs:         result = apply_hierarchy_rule(pair, rulebook)         if result.adjusted:             adjustments.append(result.log_entry)     return adjusted_queue, adjustments


3.4  Deployment Pre-Flight
The master orchestration function. Called by master_engine.py at NEUTRAL threshold detection. Runs all checks, applies all auto-resolvable adjustments, returns CLEARED / ADJUSTED / BLOCKED, and writes the complete pre-flight report to the execution log.

def deployment_preflight(queue, book, nav, regime_score):     log = PreflightLog(timestamp=now(), trigger_score=regime_score)      # Step 1: Project post-deployment thesis concentration     projected = simulate_post_deployment_concentration(queue, book, nav)     breaches  = check_cap_breaches(projected, rulebook.caps)      # Step 2: Auto-resolve cap breaches per rulebook     if breaches:         queue, adjustments = resolve_cap_breaches(queue, breaches, rulebook)         log.record_adjustments(adjustments)      # Step 3: Sizing coherence check and auto-resolution     queue, coherence_adj = check_sizing_coherence(queue, registry, rulebook)     log.record_adjustments(coherence_adj)      # Step 4: CDF signal validation (read-only from cdf.py)     cdf_alerts = validate_cdf_signals(queue)     log.record_cdf_status(cdf_alerts)      # Step 5: Detect unresolvable rule conflicts (Resolution C only)     conflicts = detect_rule_conflicts(queue, rulebook)     if conflicts:         log.set_status('BLOCKED')         engineering_ticket.create(conflicts)         return log   # Execution halted. No PM prompt generated.      log.set_status('CLEARED' if not adjustments else 'ADJUSTED')     execution_engine.execute(queue)     return log


4.  Execution States


PAIE produces exactly three execution states. No other states exist.

Status | Condition | System Response
CLEARED | All checks pass. No cap breaches. No coherence conflicts. | Queue executes automatically. Entry written to execution log.
ADJUSTED | Cap breach or coherence gap detected. Resolved by existing rulebook rule. | PAIE auto-adjusts sizing. Adjustment logged. Execution proceeds. PM reads outcome in EOD snapshot.
BLOCKED | Conflicting rules with no defined hierarchy. Resolution C condition. | Queue halted. Engineering ticket auto-generated. PM receives single EOD notification that amendment is in progress. No PM decision required.


The BLOCKED state is the only condition that generates any PM notification — a single line in the EOD snapshot confirming that a rulebook amendment is being engineered. It is not a question. It is not a request for input.

7.  Rulebook Structure


All caps, hierarchy rules, and coherence logic are stored in rulebook.json — not hardcoded in paie.py. Rules are amended through the engineering process. The PM provides thesis conviction context at intake. Engineering implements it.

8.  Explicit Scope Exclusions


PAIE does not govern, override, or interfere with:

CAM — PTRH sizing is fully autonomous and outside PAIE jurisdiction
CDF — conviction decay triggers are CDF domain; PAIE reads CDF output as signal check only
PDS — drawdown ceilings and sentiment triggers are PDS domain
Intraday rebalancing — PAIE runs at deployment trigger events, not continuously
Single-position sizing outside of deployment events — governed by existing module rules

PAIE's jurisdiction is the aggregate portfolio architecture at the moment of queue execution. It sees the book as a whole. All other modules see individual positions or individual risk dimensions.
