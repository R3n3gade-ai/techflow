0. Purpose and Design Philosophy 
The daily ARMS monitor is generated at 0530 CT from the overnight data snapshot and delivered at  0600 CT. The PM reads it at 0800 CT — 90 minutes before market open. At that point the monitor  reflects pre-market conditions and everything the system has assessed overnight. It is the right  instrument for its purpose: regime orientation, thesis status, module health, and the decision queue  entering the day. 
But markets move during the session. Earnings calls happen at 0900. Regulatory announcements drop  at 1000. CDM signals fire throughout the day. Tier 1 veto windows open and close. LAEP executes  orders. By 1455 CT — five minutes before the close — the portfolio may be in a materially different  state than the morning monitor described. 
The End-of-Day Snapshot addresses this. It is not a second full monitor. It is a focused five-field closing  status check that tells the PM and GPs exactly what changed during the session, what executed, what  is still pending, and what the regime looks like entering the overnight period. It reads in under two  minutes. It requires no action in most sessions. When it does require action it surfaces a single specific  item — not a general alert. 
The morning monitor sets the day. The End-of-Day Snapshot closes it. Together they  constitute a complete daily information architecture with no gap between the open and the  close.

2. The Five Fields — Complete Specification 
The End-of-Day Snapshot contains exactly five fields. No more. The discipline of five fields is  intentional — if a sixth field is ever proposed, the correct response is to ask whether the morning  monitor should be updated instead. The EOD Snapshot is a closing confirmation instrument, not a  second full monitor. 
Field 1 — Regime Score Delta 
Has the regime score moved materially since 0800 CT? If yes, what drove it?
Content  | Current regime label + current score · Morning score (from 0600  CT monitor) · Delta (current minus morning) · Direction arrow ·  One-sentence narrative if delta exceeds threshold
Materiality threshold  | Delta of ±0.03 or greater triggers a narrative. Below ±0.03  displays as 'Stable — no material movement during session.'

Field 4 — Session Execution Summary 
What did ARMS execute during the session? What is the current portfolio composition  versus the target?
Content  | Summary of all order executions during the session · Any  execution failures · Current portfolio weights versus Architecture  AB targets · Any weight drift above 0.5pp from target
If no executions  | Single line: 'No orders executed during session. Portfolio weights  within tolerance.'
Execution summary format  | [Time CT] [Module] [Action] [Ticker] [Size] [Price] [Status:  FILLED / PARTIAL / FAILED]
Weight drift  | Any position where current weight deviates more than 0.5pp  from target weight is flagged. Not an action item — an  information item. The system will correct at next regime cycle.
Execution failures  | Any FAILED execution is displayed in RED with the error reason.  This is the one execution field that may require PM attention — a  failed order means a target allocation is not yet achieved.

Field 5 — Overnight Risk Assessment 
What is the most important thing to watch between now and the 0800 CT morning review?
Content  | A single AI-generated two to three sentence overnight watch  statement. Not a list. Not a general market summary. One  specific, actionable statement about the highest-priority risk or  opportunity entering the overnight period.
Generation  | Claude API call via claude_wrapper.py. Input: current regime  score + delta + any open CDM/TDC signals + PTRH status +  any pending Tier 1 items. Output: two to three sentence watch  statement.
Scope constraint  | The watch statement must reference a specific named signal,  entity, or threshold. Generic statements ('markets remain  volatile') are rejected by a validation step that checks for named  specificity. If the Claude API returns a generic statement, the  system retries once with a stricter prompt before falling back to a  rule-based statement.
Prompt structure  | System: You are the ARMS risk assessment engine. Generate a  two to three sentence overnight watch statement for the PM and  GPs. Be specific — name the entity, signal, or threshold that  matters most. Do not summarize the session. Identify the single  most important thing to watch between now and 0800 CT  tomorrow. Current state: [regime + delta + signals + pending  items].
Data source  | Claude API · all module outputs at 1450 CT generation
Example — war scenario  | Watch: Iran's response to Trump's 1400 CT ultimatum is  expected via state media overnight. If Iran formally announces  Hormuz toll rates above $500,000 per vessel, the regime score  will likely cross 0.80 at the CRASH boundary. Confirm PTRH  dual-risk coverage before tomorrow's open.

3. Visual Layout Specification 
The EOD Snapshot uses the same CSS design system and color variables as the morning monitor.  The layout is a single vertical column — five cards stacked top to bottom, each self-contained. No  tables, no multi-column layout. The PM reads top to bottom in under two minutes. 
Position  | Field  | Card style  | Height  | Normal-session read time
1  | Regime Score   Delta | Full-width card.  Score bar. Delta  indicator. Color  coded   green/red/grey. | Compact —  80px | 5 seconds
2  | Pending Tier 1   Windows | Full-width card.  Green if clear.   Amber/red if   items pending   with countdown. | Compact if   clear ·   Expanded if   items present | 5 seconds if clear · 20   seconds if items present
3  | Intraday   CDM/TDC   Signals | Full-width card.  Green if clean.  Amber/red with  signal details if  signals fired. | Compact if   clean ·   Expanded if   signals   present | 5 seconds if clean · 30   seconds if signals present
4  | Session   Execution   Summary | Full-width card.  Green if no   executions or   clean   executions. Red  border if any   FAILED orders. | Medium —  execution list  if orders   present | 10 seconds
5  | Overnight Risk   Assessment | Full-width card.  Gold left border.  Two to three   sentence watch  statement.   Always present. | Fixed —  100px | 10 seconds



Header: identical to morning monitor but with session date, generation time (1450 CT), and a distinct  badge: 'END-OF-DAY SNAPSHOT — 1455 CT'. Footer: identical to morning monitor. 
CRITICAL DESIGN RULE: The EOD Snapshot must never expand to more than one printed page  when rendered as PDF. If any field generates content that would push the layout beyond one page, the  content is truncated with a 'Full detail in session log' reference. The one-page constraint is enforced at  the template level, not at the data level. The PM reads a one-page closing confirmation, not a second  full monitor.

def generate_eod_snapshot() -> EODSnapshotOutput: 
 # Field 1 — Regime Score Delta 
 morning_score = db.get_morning_monitor_score(today) 
 current_score = aras.get_current_composite_score() 
 delta = current_score - morning_score 
 narrative = generate_delta_narrative(delta) if abs(delta) >= 0.03 else None 
 # Field 2 — Pending Tier 1 Windows 
 pending_items = confirmation_queue.get_open_items()
