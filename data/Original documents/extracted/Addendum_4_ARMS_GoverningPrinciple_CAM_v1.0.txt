1. The Full Principle — Stated Precisely 
1.1 What the PM's edge is not 
The PM's edge is not analytical superiority over the algorithm. Every public signal — earnings calls,  regulatory filings, price action, macro data, options flow, job postings, patent filings — is processed  faster and more completely by ARMS than any human can read it. Attempting to out-analyze the  system is not only futile — it is a misallocation of the most scarce resource in the enterprise: the PM's  time and attention. 
Every minute the PM spends confirming what the system already knows is a minute they are not  building the network that creates the only durable edge left in modern markets. 
1.2 What the PM's edge is 
The PM's irreplaceable edge is physical presence in the rooms where the future is being decided  before it enters any dataset. 
• The lunch with the CEO who is quietly reducing their position — before the Form 4 is filed. • The conversation with the TSMC supply chain executive whose capacity allocation decisions  predict semiconductor demand two quarters ahead of any earnings call. 
• The network that surfaces the insight before it becomes a signal — and the signal before it  becomes a headline. 
• Not illegal insider information. Earned, relationship-driven awareness that precedes consensus  by weeks or months and cannot be replicated by any algorithm operating on public data. 
This is Category A intelligence. It is what SENTINEL Gate 6 was built to capture. It is the only input the  system cannot generate itself. Everything else ARMS can produce — and must produce — without  human involvement. 
1.3 The division of labor — stated as a law 
The system executes everything that can be decided by math, process, and data inflow. The  PM executes everything that requires being in the room where the data does not yet exist. 
There are no exceptions to this division. A recurring decision queue item that requires the same PM  action every day is not a safety feature. It is evidence of an incomplete build. A confirmation window for  a deterministic outcome is not discipline. It is latency. A sizing decision made by PM opinion rather than  MICS formula is not conviction. It is noise.

PART TWO — PTRH COVERAGE ADEQUACY MODEL CAM — Autonomous Tail Hedge Sizing · Addendum to Module A (PTRH Automation) 2. Why CAM Exists — The Gap It Closes 
The PTRH sizing question — how much tail protection does the portfolio need right now — is a math question. It has always been a math question. ARMS has all the inputs. The PM  should never be asked to answer it. 
Module A (PTRH Automation, specified in Addendum 1) closes the execution gap: the system rolls puts  automatically, resizes on regime transition, and corrects notional drift. What it does not do is  autonomously determine whether the current coverage level is adequate given the specific risk  configuration of the portfolio at any given moment. 
The current specification requires a PM declaration of a dual-risk event before the system applies an  override coverage standard. That declaration is a human touchpoint. It is asking the PM to answer a  math question. The CAM removes that touchpoint entirely by calculating the required coverage from  first principles using data ARMS already has.

3.2 The Coverage Requirement Formula 
The required coverage is the product of four multipliers applied to a base coverage rate. Each multiplier  reflects a specific risk dimension. When all four are elevated simultaneously — as they are today — the  system automatically increases coverage without any human involvement. 
def calculate_required_coverage( 
 current_equity_pct: float, # e.g. 0.40 for 40% equity 
 regime_score: float, # 0.0 - 1.0 
 fem_concentration_score: float, # 0.0 - 1.0 (ALERT = >0.80) 
 macro_stress_score: float, # 0.0 - 1.0 
 cdm_active_signals: int, # count of CRITICAL/HIGH signals 
 nav: float # current portfolio NAV 
) -> float:

 # Base coverage: 1.2% of NAV (minimum permanent insurance)  BASE_COVERAGE_PCT = 0.012 
 # Multiplier 1: Regime — higher score = more coverage needed  # Maps linearly from 1.0x at RISK_ON to 2.0x at CRASH boundary  regime_multiplier = 1.0 + (regime_score * 1.0) 
 # Multiplier 2: Equity exposure — more equity = more to protect  # Full deployment (100%) = 1.0x, current 40% ceiling = 0.67x floor  equity_multiplier = 0.67 + (current_equity_pct * 0.33) 
 # Multiplier 3: Concentration — correlated book = amplified loss  # NORMAL (<0.65): 1.0x · WATCH (0.65-0.80): 1.15x · ALERT (>0.80): 1.30x  if fem_concentration_score > 0.80: 
 concentration_multiplier = 1.30 
 elif fem_concentration_score > 0.65: 
 concentration_multiplier = 1.15 
 else: 
 concentration_multiplier = 1.00 
 # Multiplier 4: Acute stress — macro + CDM signals 
 # Base macro stress component 
 stress_multiplier = 1.0 + (macro_stress_score * 0.40) 
 # Each active CDM CRITICAL/HIGH signal adds 3% to stress multiplier  stress_multiplier += (cdm_active_signals * 0.03) 
 stress_multiplier = min(stress_multiplier, 1.60) # cap at 1.60x 
 # Required coverage percentage of NAV 
 required_pct = (BASE_COVERAGE_PCT 
 * regime_multiplier 
 * equity_multiplier 
 * concentration_multiplier 
 * stress_multiplier) 
 # Floor: never below regime-table minimum (from Module A)  regime_minimum = get_regime_table_minimum(regime_score) 
 required_pct = max(required_pct, regime_minimum) 
 # Ceiling: never above 3.5% of NAV (prevents over-hedging)  required_pct = min(required_pct, 0.035) 
 return required_pct * nav # returns required notional in dollars 3.3 Worked example — current conditions (March 27, 2026)

3.4 Auto-correction logic 
def run_cam_correction(required_notional: float, 
 current_notional: float, 
nav: float, 
positions: List[OptionsPosition]) -> CamResult: 
 tolerance = 0.05 # 5% tolerance band — avoid constant micro-adjustments  pct_diff = abs(required_notional - current_notional) / required_notional 
 if pct_diff <= tolerance: 
 return CamResult(action='NONE', reason='Within tolerance band') 
 if required_notional > current_notional: 
 # Increase coverage — buy additional puts at ATM 
 delta = required_notional - current_notional 
 execute_put_purchase(notional=delta, strike='ATM') 
 log_cam_action('INCREASE', delta, required_notional, current_notional)  return CamResult(action='INCREASED', delta=delta)

Note the asymmetric tolerance: the system corrects under-coverage at 5% deviation but only corrects  over-coverage at 15% deviation. This reflects the asymmetric cost of being wrong in each direction.  Under-hedged in a crash is catastrophic. Over-hedged is merely slightly expensive.
